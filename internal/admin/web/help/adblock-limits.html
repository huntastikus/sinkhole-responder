What the responder can and cannot do

Adblock-defeat explained + honest limits

Sinkhole Responder answers requests that DNS has already redirected to it. It can satisfy simple checks that only ask whether a resource loaded. Its curated JavaScript stubs also expose a few known globals so some pages do not fail immediately when an ad or analytics library is absent.

It is not a browser extension, cosmetic filter, proxy, or general HTTPS interceptor. A site-specific rule can model a small observed response, but it does not become the vendor.

Hard limits: what it cannot do

  • It does not block first-party or same-domain ads when the real hostname still resolves to the site's real server.
  • It cannot beat detection that never performs a DNS lookup through AdGuard Home: page-bundled logic, a hard-coded IP, DNS-over-HTTPS that bypasses your resolver, or a server-side check.
  • HTTPS needs deliberate CA trust on each client. Without that trust, certificate errors are correct and expected.
  • It cannot prove an ad impression to a vendor, mint vendor cookies or tokens, reproduce signed content, or satisfy arbitrary origin, credential, signature, or response checks.
  • It cannot satisfy Subresource Integrity (SRI) hashes for the original resource or bypass certificate pinning.
  • It cannot emulate QUIC/HTTP-3-only behavior. The responder serves TCP HTTP/1.1 and HTTP/2.
  • Browsers may classify a public hostname that resolves to a private LAN address as local-network access, then prompt or block the request according to browser policy.

What success looks like

A redirected request receives a safe, valid response and the page continues when its checks are simple enough. Deep render verification or a server-gated “disable adblock to continue” flow can still fail.

The responder also does not remove page layout. An empty ad box can remain after its resource was replaced; that is expected. See Troubleshooting or pair the responder with a browser cosmetic filter if you control the client.

Test the actual browser and site. Browser versions and application-specific checks change, and this service cannot override client security policy. For HTTPS boundaries, see TLS and trusting the CA.
