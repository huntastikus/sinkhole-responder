Operator checklist

Security and hardening

This service accepts requests carrying attacker-controlled hostnames, paths, and headers. Keep the deployment narrow even on a home or lab network.

Run with least privilege

  • Run the container as non-root UID/GID 65532.
  • Drop all Linux capabilities, then restore only NET_BIND_SERVICE when binding ports 80 and 443.
  • Use a read-only root filesystem, no-new-privileges, and only the state directory as writable storage.
  • Keep CA private keys and other private keys owner-readable only, normally mode 0600 or stricter.

Constrain the network

  • Allow public responder ports only from trusted LAN subnets.
  • Keep management /metrics and /healthz on the default loopback listener. Never expose port 9090 publicly.
  • The responder needs no initiated outbound traffic. Enforce a suitable no-egress policy where your runtime supports it, while still allowing response traffic back to trusted clients.

Use the hardened Compose or systemd example supplied with the project as your baseline. The deployment README's Security hardening section includes LAN firewall examples and the operational details behind this checklist.

Protect the local CA

A client that trusts this CA accepts certificates it signs for any HTTPS hostname. Use it only on controlled lab/home devices, never distribute it, and remove trust when the experiment ends.
