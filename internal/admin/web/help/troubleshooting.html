Troubleshooting

Is traffic even reaching this box?

  1. Watch the dashboard request counter while loading the blocked resource. If it does not increase, the live request probably did not reach the responder.
  2. Use Tools → Test a domain to confirm which saved rule, rulepack, or default response would win. This is a dry run; it does not prove that client traffic used this DNS path.
  3. Bypass DNS and test the responder directly: curl -v --resolve blocked.example:80:RESPONDER_IP http://blocked.example/banner.js
  4. Check the system-health banner for listener, TLS, state-directory, recent-error, or rulepack warnings.

DNS is not redirecting the blocked name

Run dig blocked.example @ADGUARD_IP and confirm the answer is the responder's LAN address. Then check AdGuard Home's Custom IP fields, browser Secure DNS / DNS-over-HTTPS, operating-system DNS, and VPN DNS. Follow the AdGuard Home setup guide.

The browser shows a certificate error

For local-CA mode, download the public CA and deliberately trust it on that client. For static mode, make sure the certificate covers the original hostname. Do not make curl -k or disabled browser verification the normal fix. Follow TLS and trusting the CA.

An empty ad box is left over

This is expected. The responder returns a harmless placeholder for a redirected resource; it does not edit the page, remove first-party layout, or hide same-domain ad slots. Read Adblock-defeat and honest limits for the boundary.
