Response selection

Rules and rulepacks

Rules let a particular request receive a more useful placeholder than the general default. They are evaluated from top to bottom, and the first match wins.

Create a rule

Open the rule builder and work top to bottom:

  • Add rule opens the editor. Give the rule a Name — a short label shown in the rule list and the preview.
  • Add at least one matcher. A rule with no matcher is rejected, and every matcher you fill in must match the request (logical AND).
  • Choose the response: an Embedded asset (a built-in, media-safe body whose content type is set for you) or a Custom body with your own text and Content-Type.
  • Optionally set a Status, a Delay, and extra response headers.
  • Apply rule adds it to the list. Drag a rule's handle to reorder — position decides precedence, because the first match wins.
  • Save rules writes the change and live-reloads the responder; no restart is needed.
  • Run the Live preview with a sample host, path, method, destination, and Accept to see which rule wins (or the generic fallback) and the exact status, content type, delay, and body. Preview reflects saved rules, so save first.

Match fields

Fill only the matchers you need; each one you set must match, and a rule needs at least one. Your own rules always run before rulepack rules.

  • Method — exact HTTP method (GET, HEAD, POST, PUT, PATCH, DELETE, OPTIONS), or leave as Any.
  • Host — exact hostname with no scheme or path, case-insensitive. Prefer this over a glob when you can.
  • Host glob — shell-style hostname pattern such as *.example.com.
  • Path glob — shell-style URL-path pattern such as /ads/*; the query string is never matched.
  • Path regular expression — a Go regular expression matched against the URL path, such as ^/ad(s|server)/.
  • Sec-Fetch-Dest — the browser request destination, such as script, image, style, or iframe.
  • Accept contains — case-insensitive text that must appear in the request Accept header.
  • Query matchers — a parameter name plus an optional exact value; an empty value matches when the key is merely present.
  • Request header matchers — a header name (case-insensitive) plus an exact value.

Response fields

A matched rule overrides only the response parts you set; anything left blank falls back to the generic selection.

  • Embedded asset — one built-in body: transparent-gif, transparent-png, transparent-svg, empty-js, empty-css, empty-json, blank-html, empty-text, silent-wav, or minimal-mp4. Its content type is set automatically.
  • Custom body — your own literal text plus an optional Content-Type (left empty, a type is inferred from the request). A rule uses either an embedded asset or a custom body, never both.
  • Status — an HTTP status from 100 through 599, or 0 to keep the configured default.
  • Delay — an artificial wait of 0 through 10000 ms before responding.
  • Extra response headers — added after the standard defaults. Set-Cookie is always dropped and bodies are stripped for 204/304; avoid overriding Content-Length or the security headers.

For base64 or file-backed bodies, edit the raw YAML under Configuration: body_base64 decodes a binary body and body_file loads a file (confined to the config directory, 1 MiB maximum) at each save or reload.

Rulepacks

A rulepack is a built-in, versioned collection of rules for a known script or ad network. Each pack only rewrites specific script URLs to a harmless stub and never fetches anything at runtime. Your own rules always evaluate before rulepack rules, so a user rule can override a pack; within each ordered group, first match still wins. Enable or inspect packs on the Rulepacks page.

Enabling the recommended pack turns on these five high-confidence packs for the most common ad and tracking networks, each matched only on the vendor's own hostnames:

  • Google AdSense — neutralizes the AdSense display-ad scripts (adsbygoogle.js, show_ads) on pagead2.googlesyndication.com with a no-op stub.
  • Google Publisher Tag — replaces GPT (gpt.js) on securepubads.g.doubleclick.net and www.googletagservices.com with an API-compatible stub.
  • Google Analytics — neutralizes the Analytics and Tag Manager loaders (analytics.js, ga.js, gtag/js, gtm.js) on google-analytics.com and googletagmanager.com.
  • Meta Pixel — neutralizes the Pixel loader and signal configuration (fbevents.js, signals/config) on connect.facebook.net.
  • Amazon TAM — neutralizes the Amazon Publisher Services loader (apstag.js) on c.amazon-adsystem.com.

These packs are not part of recommended; enable them individually when you need them:

  • Consent Management Platforms — neutralizes the OneTrust consent loaders (otSDKStub.js, otBannerSdk.js) on cdn.cookielaw.org.
  • Prebid — neutralizes Prebid scripts (prebid*.js). Matched on any host because Prebid is frequently served first-party, so it is broad and may over-match.
  • Anti-Adblock Detectors — neutralizes common detection scripts (fuckadblock*.js, blockadblock*.js). Matched on any host and best-effort by nature.

A matching rule can imitate a small response shape, not a complete vendor service. Read Adblock-defeat explained + honest limits before building site-specific expectations.
